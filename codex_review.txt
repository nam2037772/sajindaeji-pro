정적 리뷰 기준으로 보이는 버그·엣지케이스·잠재 문제입니다.

1. **PDF 본문 페이지가 세로로 늘어날 가능성**
   `.print-page`는 `210mm x 277mm`인데, PDF에는 `210mm x 297mm`로 `addImage` 됩니다. 본문 캔버스가 A4 전체 높이에 맞춰 강제로 늘어나서 사진·표·여백이 세로로 왜곡될 수 있습니다.

2. **본문 하단 페이지 번호가 내용 위에 겹칠 수 있음**
   `doc.addImage(..., 0, 0, pageWidth, pageHeight)`로 페이지 전체를 이미지로 채운 뒤 `pageHeight - 6` 위치에 페이지 번호를 덧그립니다. 본문 이미지 안에 별도 하단 여백이 보장되지 않아 마지막 사진 카드나 표와 겹칠 수 있습니다.

3. **사진이 많은 경우 PDF 생성 중 메모리 폭증 가능**
   `buildPrintPreview()`가 모든 페이지와 모든 base64 이미지를 한 번에 DOM에 만들고, 각 페이지를 `scale: 2` 캔버스로 캡처한 뒤 `toDataURL()`까지 생성합니다. 사진 수가 많으면 브라우저가 멈추거나 탭이 죽을 수 있습니다.

4. **`toDataURL()` 사용으로 메모리 사용량이 더 커짐**
   캔버스 이미지를 JPEG data URL 문자열로 만든 뒤 jsPDF에 넣고 있습니다. base64 문자열은 Blob보다 메모리 부담이 큽니다. 여러 페이지 PDF에서 특히 문제가 됩니다.

5. **오프스크린 렌더링 위치가 너무 멀어 캡처 실패 가능성**
   PDF 캡처용 DOM을 `left: -99999px`에 배치합니다. 브라우저·html2canvas 조합에 따라 화면 밖 fixed/offscreen 요소 캡처가 빈 화면이 되거나 레이아웃 계산이 불안정할 수 있습니다.

6. **PDF 생성 중 사용자가 데이터를 수정하면 결과가 불안정할 수 있음**
   PDF 버튼만 비활성화되고 입력, 삭제, 이동, 사진 변경은 계속 가능합니다. 생성 중 `render()`가 실행되면 `printPreview`가 다시 만들어져, `pages` 배열이 가리키는 기존 DOM이 분리되거나 캡처 실패·누락이 생길 수 있습니다.

7. **이미지 디코딩 완료를 명시적으로 기다리지 않음**
   `buildPrintPreview()` 후 `requestAnimationFrame` 두 번만 기다립니다. 대부분은 동작하겠지만, 큰 이미지나 느린 환경에서는 `<img>` 디코딩이 끝나기 전에 html2canvas가 캡처를 시작해 빈 이미지 또는 누락이 생길 수 있습니다.

8. **빈 항목도 총 사진 수에 포함됨**
   표지의 `총 사진 수`는 `state.items.length`입니다. 이미지가 없는 빈 항목도 “사진 수”에 포함되어 사용자 기대와 다를 수 있습니다.

9. **검색 중 PDF 저장 시 검색 결과가 아니라 전체 항목이 출력됨**
   화면 목록은 검색 필터가 적용되지만 `buildPrintPreview()`는 항상 `state.items` 전체를 사용합니다. 현재 보이는 항목만 저장된다고 기대하면 혼동될 수 있습니다.

10. **파일명 sanitize가 PDF 저장에는 적용되지만 JSON 일반 저장에는 부족함**
   PDF 파일명은 `sanitizeFileName()`을 거치지만 `exportJson()`은 `reportDate`만 쓰고, `exportJsonSaveAs()`의 prompt 입력값은 금지 문자를 정리하지 않습니다. 일부 OS/브라우저에서 저장명이 깨질 수 있습니다.

11. **`saveBlobAs()` 폴백 다운로드가 Blob을 다시 Blob으로 감쌈**
   `download(defaultName, blob, blob.type)` 호출 후 `download()` 내부에서 `new Blob([content])`를 만듭니다. 동작은 가능하지만 불필요한 래핑이고 큰 PDF에서는 메모리 부담이 됩니다.

12. **localStorage 저장 실패 처리가 제한적**
   `save()`는 `QuotaExceededError`만 사용자에게 알립니다. Safari private mode, 보안 설정, 스토리지 차단 등에서 `SecurityError`가 나면 조용히 실패할 수 있습니다.

13. **사진 저장 구조상 localStorage 한계에 빨리 도달**
   업로드 이미지를 base64 JPEG로 `state.items`에 넣고 localStorage에 저장합니다. 사진 몇 장만 많아져도 5~10MB 제한에 걸릴 수 있고, PDF 기능 추가와 별개로 백업 전 데이터 유실 위험이 큽니다.

14. **업로드 변환 과정에서 투명 PNG/WebP가 손상될 수 있음**
   모든 이미지를 JPEG로 변환합니다. 투명 배경 이미지는 검정색 또는 의도치 않은 배경으로 합성될 수 있습니다.

15. **EXIF 방향 정보 관련 엣지케이스**
   `canvas.drawImage(img, ...)` 기반 리사이즈는 브라우저에 따라 EXIF orientation 처리 결과가 다를 수 있습니다. 일부 사진이 회전되어 저장·PDF 출력될 가능성이 있습니다.

16. **가져온 JSON 데이터 검증이 거의 없음**
   `items`가 배열인지 정도만 확인합니다. `image`에 매우 큰 문자열, 이상한 URL, 잘못된 필드 타입이 들어오면 렌더링/PDF 생성이 느려지거나 실패할 수 있습니다.

17. **`공종` 필드가 내부적으로 `memo`에 저장됨**
   기능상 동작은 하지만 UI의 “공종”과 데이터 필드명 `memo`, 프로젝트 메모 `projectMemo`가 섞여 있어 추후 JSON 호환성이나 유지보수에서 실수하기 쉽습니다.

18. **PDF 표지와 본문 페이지 크기 기준이 다름**
   표지는 `297mm` 높이, 본문은 `277mm` 높이입니다. 본문이 인쇄 CSS의 여백 기준에서 온 값처럼 보이는데, PDF에서는 전체 A4로 넣고 있어 표지와 본문 스케일 감각이 달라질 수 있습니다.
